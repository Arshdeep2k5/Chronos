# Chronos Core Protocol (CCP) Specification

*This document defines the Cognitive Kernel (Layer -1) of the Chronos Personal Context Operating System (PCOS). It establishes the universal runtime objects, lifecycle contracts, and ownership boundaries that all subsystems must obey. Before any feature is built, it must conform to these canonical structures.*

## 1. The Governing Principle
**Chronos exists to preserve and extend human cognitive continuity.**
Every event, state, decision, and action must ultimately serve this goal.

---

## 2. The Semantic Hierarchy

Knowledge in Chronos is no longer flattened into generic "Context Nodes." It flows upward through five strict semantic levels:

1. **Observation:** A raw, objective, atomic event recorded by an adapter. (e.g., *Opened `main.py`*, *Copied text*, *Switched git branch*).
2. **Artifact:** A concrete, addressable object derived from observations. (e.g., *The file `main.py`*, *A WhatsApp message thread*).
3. **Concept:** An understood grouping of artifacts. (e.g., *The "Chronos Knowledge Graph"*).
4. **Intent:** The inferred objective the user is actively pursuing. (e.g., *Implement Semantic Graph*). **This is the center of the graph.**
5. **Commitment:** A formal obligation or deadline tied to an Intent. (e.g., *Finish Graph Engine before Sunday*).

---

## 3. Cognitive Sessions

Humans do not experience work as a stream of isolated events; they experience work in **Sessions**. 

A Cognitive Session bounds work by time, Intent, and context. The Context Continuation Engine (CCE) does not simply restore tabs or files; it restores *Cognitive Sessions*.

**Session Structure:**
- **Intent:** The inferred objective driving the session.
- **Duration:** Time spent actively in the session.
- **Observations:** Count of raw events generated.
- **Artifacts:** The files, tabs, and documents touched.
- **Concepts:** The overarching topics explored.
- **Decisions:** The autonomous interventions triggered by Chronos.
- **Outcome:** The result (e.g., *Completed, Abandoned, Interrupted, Paused*).
- **Resume Token:** A serialized state package allowing instantaneous restoration of this exact cognitive context.

---

## 4. Canonical Runtime Objects

Every subsystem in Chronos communicates exclusively using one of these six universal runtime objects over the **Cognitive Bus**. 

### 1. `ChronosEvent`
- **Definition:** The fundamental unit of perception. 
- **Ownership:** Generated by Layer 1 (Perception Adapters).
- **Contract:** Must be immutable, strictly factual, and contain no inference.
- **Payload Example:**
  ```json
  {
    "id": "evt_9832",
    "type": "BranchChanged",
    "source": "GitAdapter",
    "timestamp": "2026-06-27T10:15:00Z",
    "payload": {
      "repository": "/Users/chronos",
      "previous_branch": "main",
      "new_branch": "feature/cognitive-bus"
    }
  }
  ```

### 2. `ChronosState`
- **Definition:** The current, aggregated world model of the user's environment.
- **Ownership:** Maintained by Layer 2 (Memory) and updated via the Cognitive Bus.
- **Contract:** Represents the absolute ground truth at any given millisecond. Used by Prediction engines to forecast outcomes.

### 3. `ChronosIntent`
- **Definition:** The active objective inferred from the `ChronosState`. 
- **Ownership:** Generated by Layer 3 (Reasoning / Intent Engine).
- **Contract:** All active commitments and artifacts must map to a `ChronosIntent`. When an Intent shifts, the active Cognitive Session ends, and a new one begins.

### 4. `ChronosDecision`
- **Definition:** A formal resolution to intervene, alert, or modify the workspace.
- **Ownership:** Exclusively produced by Layer 4 (Decision Orchestrator).
- **Contract:** No other subsystem (e.g., ODE, DDE, PCM) can generate a Decision. They may only submit *Evidence*. The Orchestrator resolves conflicts and issues the Decision.

### 5. `ChronosAction`
- **Definition:** An executable command translating a Decision into a physical system change.
- **Ownership:** Consumed and executed by Layer 5 (Action Executor).
- **Contract:** Actions must be idempotent, safely reversible (where possible), and must pass through the Layer 6 Consent Gate if they affect external environments.

### 6. `ChronosReflection`
- **Definition:** The learning record generated after an Action concludes (or is ignored).
- **Ownership:** Generated by Layer 3 (Reflection Engine).
- **Contract:** Compares the initial Prediction against reality to adjust the weights, thresholds, and confidence scoring of the Personal Capacity Model (PCM) and Decision Orchestrator.

---

## 5. The Universal Lifecycle

By enforcing the six canonical objects, Chronos transitions from a web of fragile point-to-point API calls into a robust, executable pipeline:

`Reality → Observe → Interpret → Knowledge Mutation → Prediction Update → Decision Trigger → Execution Request → Reflection Record`

**Example Execution Flow:**
1. **Git Adapter** observes reality and publishes a `ChronosEvent` (BranchChanged).
2. **Cognitive Bus** distributes the event.
3. **Intent Engine** interprets the event, infers the user is starting new work, and mutates the `ChronosState`, generating a new `ChronosIntent`.
4. **Deliverability Predictor (DPE)** detects the new Intent, recalculates the schedule, and publishes a forecast.
5. **Decision Orchestrator** evaluates the forecast. It generates a `ChronosDecision` to suggest staging related research files.
6. **Action Executor** converts the Decision into a `ChronosAction` and displays a prompt on the HUD.
7. **Reflection Engine** records whether the user accepted the staging, generating a `ChronosReflection` to improve future suggestions.

---

*This specification represents the permanent Core Contract of Chronos. Adapters, UI widgets, and reasoning engines can be swapped, added, or deleted, but they must all speak this language.*
